## 快速开始

```rust
// 主入口
src-tauri/src/
├── lib.rs          # 模块导出、命令注册
├── main.rs         # 程序入口
├── api/            # DTO / 共享类型
├── core/           # 核心业务逻辑
│   ├── dbi/        # 🔥 统一数据访问层（总调度）
│   ├── driver/     # 驱动核心：native + JDBC + wasm 统一管理
│   ├── services/   # 业务服务：SQL执行、连接管理
│   └── datasource/ # 数据源管理：注册/路由
├── adapters/       # 适配器层（Tauri）
└── commands/       # 命令模块
```

## 核心原则

1. **分层架构**：严格分层，上层依赖下层，禁止反向依赖
2. **无框架依赖**：Core 层不依赖任何外部框架
3. **可测试性**：所有业务逻辑可独立单元测试
4. **可扩展性**：新增数据库只需实现 Driver trait
5. **DBI 统一入口**：所有数据操作通过 DBI 接口，支持智能路由

## 关键设计决策

### 1. 为什么使用分层架构？

- **可测试性**：Core 层无框架依赖，可独立测试
- **可替换性**：适配器层可替换（Tauri → CLI/HTTP）
- **关注点分离**：每层职责单一

### 2. 为什么 Core 层不依赖 Tauri？

- 业务逻辑与框架解耦
- 便于单元测试（无需 mock Tauri）
- 未来可支持其他平台（CLI、WASM 等）

### 3. 为什么使用 Trait 抽象数据库驱动？

```rust
// 统一接口，不同实现
trait Database {
    async fn query(&self, sql: &str) -> Result<QueryResult>;
    async fn list_databases(&self) -> Result<Vec<String>>;
    async fn list_tables(&self, database: &str) -> Result<Vec<SchemaObject>>;
}
```

### 4. 为什么引入 DBI 层？

- **统一入口**：所有数据操作通过 DBI::query() / execute()
- **智能路由**：根据查询类型自动选择最佳执行引擎
- **DuckDB 加速**：复杂查询自动路由到 DuckDB 加速执行
- **会话管理**：统一的会话、事务、上下文管理
