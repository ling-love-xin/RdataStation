## 📖 这是什么？

RdataStation 是一个**本地优先、不上云**的数据库桌面工具。

市面上的数据库管理工具已经非常成熟。DBeaver是开源社区的瑰宝，DataGrip是SQL编辑体验的标杆，Navicat以稳定易用著称。我们对此充满敬意。

RdataStation的诞生，不是要取代这些伟大的产品，而是尝试回答一个它们因历史架构原因而尚未聚焦的问题：

**当用户执行完一条SQL，拿到几十万行结果集后，下一步该怎么办？**

导出到Excel？导入Python写pandas？贴到Jupyter里绘图？这个过程不仅割裂，而且在大数据量下会迅速崩塌。

**我们的答案：让分析发生在查询之后，在工具内部完成。**

---

## 👀 一个熟悉的场景

你写了一条SQL，查出来20万行数据。

你想按地区做个汇总。导出CSV，Excel打不开。切到Python，import pandas，加载CSV，写groupby。笔记本风扇呼呼转。

你只是想对查询结果多追问几个问题而已。

**在一台普通笔记本上对50万行数据的测试中：** Excel 加载耗时超过30秒，筛选时出现卡顿。Python + pandas 加载约需8秒，分组聚合约需3秒。RdataStation + DuckDB 返回相同的分组结果约在0.3秒内完成。*（测试环境为标准配置笔记本，实际结果可能因环境而异。）*

**在RdataStation里：** 在结果集面板上，点一下 **"DuckDB分析"** 按钮。20万行结果在本地DuckDB里变成一张临时表。输入聚合SQL，毫秒级返回结果。不需要导出，不需要Excel，不需要Python。整个追问过程，在一个工具里完成。

- 这张临时表是**会话级**的——关闭面板即自动清理
- 如果想把分析成果留下来，点**"持久化"**——数据存入项目级DuckDB
- 如果整个团队都需要，点**"提升为公共资产"**——存入应用级DuckDB，所有项目都能引用

## 🎯 它是什么，不是什么

RdataStation聚焦于**查询之后、可视化之前**这个分析地带。

连接、查询、管理是**底座**。二次分析、联邦查询、数据探查是**延伸**。图表、仪表盘、报告——这些交给插件生态。

我们不是在做"什么都有的数据库工具"。我们在做的是"查询结果到分析结论之间的那座桥"。

## 💡 设计哲学

### 🏠 本地优先，数据主权在用户手中

所有数据库连接信息、查询历史、分析缓存只存在于你的电脑上。没有遥测，没有云同步，没有"登录才能使用"。你的数据库密码、查询历史、分析成果，永远不会离开你的电脑。

### 🧩 SQL是入口，但不必是终点

RdataStation以SQL为分析入口——连接、查询、二次分析、联邦查询、数据探查，都通过SQL完成。当分析结果需要进一步加工时，插件系统可以接管：用Python绘制图表、用Extism插件做数据脱敏或格式转换。SQL负责"分析什么"，插件负责"产出什么"。两者在同一个工作台内协作。

### 🧠 DuckDB 作为分析引擎

DuckDB 在 RdataStation 中不是"另一个可连接的数据源"，而是嵌入产品核心的分析引擎。

- **二次分析**：任何查询结果集，可以一键转入本地 DuckDB 临时表。`GROUP BY`、窗口函数、子查询——毫秒级完成，不需要导出到外部工具。
  - *什么时候需要它：* 你刚跑完一条查询，不想导出、不想等 Excel 加载，就想在工具里继续追问。
- **联邦查询**：一条 SQL 关联 MySQL 的表、PostgreSQL 的视图和本地的 CSV 文件。
  - *什么时候需要它：* 用户表在 MySQL，订单表在 PostgreSQL，对照文件在桌面上。一条 SQL 联起来。
- **数据探查**：点击结果表格中任一列，侧边面板即时展示该列的数据画像。数值列显示均值、最小值、最大值、中位数；文本列显示去重计数、空值率、频率分布 Top 10。所有计算由本地 DuckDB 在毫秒级完成，不触动远程数据库。
  - *什么时候需要它：* 在分析之前，你想快速了解数据长什么样——有没有空值？分布如何？一键出答案。
- **三种结果集模式**：即时过滤（前端，零延迟）、SQL 过滤（拼接 WHERE 重新查询数据库，类似 DBeaver）、DuckDB 深度分析（聚合、窗口函数、联邦查询）。
- **版本快照**：公共维度表更新时，旧版本保持为快照不被覆盖。每个项目锁定到特定版本。新版本发布后，各项目自主决定何时同步。分析结果始终可复现。
  - *什么时候需要它：* 你在1月用"产品分类 v1"表做了一份季度报告，3月有人把它更新成 v2。没有版本快照，你的报告会悄悄变化。有了快照，它保持原样。

### 🏗️ 双层架构：应用级 + 项目级

RdataStation 将数据工作组织为两层，每层都有各自的 SQLite 持久化存储和 DuckDB 分析引擎。

- **项目级（个人分析沙箱）**：每个项目拥有独立的 SQLite 和 DuckDB 文件，项目之间物理隔离。你的分析过程、中间结果、探索性查询保持独立，互不干扰。
- **应用级（公共服务层）**：存放公共连接模板、主数据、维度表和分析资产。所有项目都可以引用。

当同一份数据被多个项目反复使用——比如供应商主数据被营销分析、库存分析、采购分析三个项目同时需要——用户可以**手动将其提升**到应用级公共服务层。一次清洗，处处可用。当主数据更新时，新版本作为快照创建。每个项目独立决定何时同步新版本，或锁定旧版本。

### 🧩 插件系统（设计中）

RdataStation 提供两种扩展通道：

- **Extism 轻量插件（WASM 沙箱）**：社区开发的插件，适合 SQL 格式化、数据脱敏、代码生成、分析模板。沙箱运行，确保安全。支持 Rust、Go、Python、JavaScript 等多语言开发。
- **Sidecar 重量依赖**：独立进程管理，适合 JDBC 桥接（需要 JVM）和 Python 分析环境（需要 pandas 等）。按需启动，空闲时完全不占用资源。Sidecar 进程崩溃不影响主程序。

```bash

git clone https://github.com/ling-love-xin/RdataStation.git
```

如果你需要一个稳定可靠的数据库工具，DBeaver、DataGrip、Navicat、TablePlus都是经过多年打磨的优秀选择。

如果你认同RdataStation的设计方向——让分析发生在查询之后、项目间隔离但共享公共资产、数据完全留在本地——欢迎关注、试用、提建议，或一起参与。

🔍 我们的不同之处
RdataStation 不是要做一个"更好的 DBeaver"或"更好的 DataGrip"。这些工具在自己的领域已经足够优秀。RdataStation 只是聚焦在了工作流中一个不同的环节。这不意味着哪种做法更正确——只是不同的起点，服务于不同的需求。
